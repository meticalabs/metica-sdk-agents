# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

This is **not application code** — it's a Claude Code *plugin* shipping three Metica sub-agents (compat-checker, integrator, validator) for Unity integration plus an `ad-log-monitor` skill for on-device runtime verification. No build step, no compiled artifact: changing an agent or skill = editing its `.md`; scripts change only when the system tooling does. The source:

- **Agents** — markdown-with-frontmatter directly under `agents/` (`unity-{integrator,validator,compat-checker}.md`); frontmatter is the contract, body is the prompt. They live flat in `agents/`, **not** a subfolder — a subfolder becomes a scope segment in the mention token (`agents/unity/…` → `@agent-metica-sdk-agents:unity:unity-integrator`), breaking the documented `@agent-metica-sdk-agents:unity-integrator`.
- **Skills** — `skills/<name>/SKILL.md` (auto-discovered; frontmatter is just `name:` + `description:`). Invoked via the `/` slash form or a description-matched phrase — **not** `@`-mention (agents only).
- **Scripts** — `scripts/*.sh` hold only what an agent can't do in prose: locate the plugin root, download/import the SDK, capture device logs, launch the Unity compiler, the SessionStart update check. All verification/checking logic lives in agent prose.
- **Hooks** — `hooks/hooks.json` (auto-discovered). One `SessionStart` hook → `check-for-update.sh`, which notifies only on a strictly-newer version. **Fail-open** (any uncertainty exits 0 silently). Opt out with `METICA_SKIP_UPDATE_CHECK=1`.
- **Data + templates** — `metica-versions.yaml` (compat matrix), `references/` (API-parity docs + `3pa-forwarders.md`), `agents/contracts.md` (inter-agent JSON contracts), and `scripts/templates/standalone/MeticaAdService.cs.tmpl` — all read directly by the agents.

## Commands

```bash
bash tests/run-all.sh                      # full suite; prints "ALL GREEN" or "FAILURES"
bash tests/run-<suite>-tests.sh            # a single suite: resolver, download,
                                           # compile, update-check, log-monitor
```

Tests cover the **scripts** (system-tooling plumbing) — five suites of plain bash assertions
against `tests/fixtures/`, no framework. The verification logic lives in agent prose (reviewed at
run time), not tests. The `download` suite skips silently when its local SDK build is absent; the
other four always run, and `update-check` drives `check-for-update.sh` through a fake `curl` (no
network).

## Architecture

**Orchestrator + two reasoning sub-agents.** `unity-integrator` (opus) is the only agent users invoke directly; it calls `unity-compat-checker` (haiku) and `unity-validator` (opus) as sub-agents. Both are **reasoning agents, not script-wrappers**: the compat-checker judges each compatibility check against `metica-versions.yaml`; the validator reads the project's code and reasons about every rule in one pass, citing the lines that prove each verdict — reading the vendored SDK source (`Assets/MeticaSdk/Runtime/...`) when a verdict turns on SDK behavior, and separating proven findings from hypotheses (`confidence` / `unresolved`). The validator's only shell-out is the Unity batch compile (`compile-check.sh`). Both run in a fresh context. The rules and JSON shapes live in `agents/contracts.md`; the behavior in `agents/unity-validator.md` and `unity-compat-checker.md`.

**Sub-agents signal the orchestrator in fenced JSON.** Each ends with one ` ```json ` block — the orchestrator parses the **last** one (regex in `agents/contracts.md`) — carrying a stable `"schema"` name (`compat-checker`, `validator`). The integrator emits no JSON. **Change a sub-agent's JSON output → update `agents/contracts.md` in the same commit.** That is the whole discipline.

**`PLUGIN_DIR` resolution is mandatory.** Every agent's (and the skill's) first bash command is a loop that locates and runs `scripts/resolve-plugin-dir.sh`. It is **deliberately duplicated byte-identical across all four entry points** (three agents + `skills/ad-log-monitor/SKILL.md`) and can't be DRY'd — its whole job is to *find* the scripts before any path to them is known (the bootstrap paradox); **change all four copies in lockstep.** `$CLAUDE_PLUGIN_ROOT` is not reliably exported into an agent's bash env (verified empirically), so it's only the first candidate; the load-bearing fallback is the resolver self-locating from its own script path. `is_root()` keys on `.claude-plugin/plugin.json` + an `agents/` directory — moving the manifest breaks it (and marketplace install). Covered by `tests/run-resolver-tests.sh`.

**The integrator's flow is discover → adapt → validate → autofix.** Discovery (Steps 2 + 2.5) derives whether MaxSDK is present (`HAS_MAX`) and inventories direct call sites, the wrapper, formats, placements, and per-format behaviour; `HAS_MAX` selects the mediation argument (`MAX` vs `null`), whether the game's *direct* `MaxSdk.*` call sites get rewritten, and whether the Step 3 plan asks for `MAX_MODE` — `replace` (default; call sites rewritten to Metica outright) or `switchable` (same rewrites, but each original Max call survives as the `else` branch behind the generated `MeticaAdService.UseMetica` switch, via the `__MAX_SWITCH__` token) — generated artifacts are otherwise identical, and dedicated Max-wrapper files are left untouched. When Max is present it also detects the remote-config provider, **report-only** (drives the Step 7 cohort-gating recipe; under `switchable` the recipe drives the switch). The run writes a checkpoint log at `$PROJECT/.metica-integration.log` (decisions, files written, step completion) — the audit record for autofixes and the resume point when a run is interrupted mid-flow. **Generated Metica code is a single `MeticaAdService` MonoBehaviour** with one `// @fmt-begin:<format>`…`// @fmt-end:<format>` region per used format, wrapped in the project's namespace (never `Metica.AbTest`) and landing under the detected adapter folder. **Init is CMP-aware** via the `__START_HOOK__` token: with no CMP it auto-inits with configured privacy defaults; with a CMP it is emitted empty and the host drives the consent-gated `Initialize(hasUserConsent, doNotSell)` overload after consent resolves (the validator's `init_gated_on_cmp` rule enforces this). The one template is `scripts/templates/standalone/MeticaAdService.cs.tmpl`; the discover/adapt mechanics live in `integrator.md` Steps 2.5 and 5 — edit those, not this file.

**Fresh MeticaSDK install is enforced, never performed; an SDK upgrade is auto-swapped.** The compat-checker's `metica_sdk` row reads the installed version and FAILs when missing or below target. The integrator branches on `detected`: **missing** → it only *offers* a fresh install (`download-metica-sdk.sh` helper; the user imports the `.unitypackage` and re-runs); **below target** → after the git snapshot it clean-swaps to the target (`download-metica-sdk.sh --clean --import`) and migrates the integration code. The clean-swap is the one place the integrator performs an install, gated on an existing-below-target install + the recoverable snapshot.

**`references/metica-sdk-migration.md` is the single source of truth for what changes between MeticaSDK versions.** Read by the integrator (upgrade migration, `integrator.md` Step 5) and the validator (`metica_deprecated_api`, ADVISORY). Each symbol is classed unchanged / behavior-changed / obsoleted / signature-changed / new-optional; extend the matching section on each SDK bump.

**`references/max-metica-api-map.tsv` is the single source of truth for the MaxSdk ↔ MeticaSdk surface.** Read by both the validator (`max_api_use_metica`, `max_api_unsupported`) and the integrator (Step 5 rewrite/drop), so they can't drift. Rows are `<pattern>\t<replacement>\t<kind>\t<notes>` (`kind` ∈ `rename` / `signature-change` / `drop` / `exempt`). Its narrative twin is `references/max-vs-metica-2.4.0-api.md` — **keep both in sync on an SDK bump.**

**`references/3pa-forwarders.md` is the single source of truth for the 3PA forwarder shapes and the `MeticaAds.RevenueCallbackDelivery` delivery-mode selection** (how the mode is matched to the game's MAX callback threading). Read by the integrator (Step 5 `__REVENUE_DELIVERY__` token + 3PA pass) and the validator (`threepa_forwarder_in_revenue_paid`); the migration doc, `metica-versions.yaml`, api-map, template header, and `contracts.md` all **point here** rather than restating the rule — change it in one place.

**The validator's `compiles_cleanly` rule actually builds the project** via `scripts/compile-check.sh` (Unity batch-mode — the only compiler that sees Unity's assemblies/asmdefs/defines; we deliberately do **not** fall back to `csc`/`dotnet`). One FAIL per `error CS####` with file:line — the catch-all for compile bugs, so resist re-adding bespoke per-bug string rules in the prose. **On by default**, self-skips to a non-blocking `WARN` when no Unity is located (`UNITY_PATH` overrides) or `METICA_SKIP_COMPILE=1` is set. Covered by `tests/run-compile-tests.sh`.

**The `ad-log-monitor` skill verifies a Metica + AppLovin MAX integration from live device logs.** It is a **skill, not an agent**, because the workflow is interactive (start → play → stop → analyse → repeat → compare) and benefits from staying in the main conversation. Four beats: **1** (`log-monitor-start.sh`) gates the toolchain, detects platform, and launches a background capture claiming a `<label>.session`; **2a** (`log-monitor-stop.sh`) kills the capture and prints a summary — no analysis; **2b** (prose) reads the log and writes `<label>-analysis.md`; **3** (prose) compares trial vs holdout and writes `compare-trial-vs-holdout.md`. **2b and 3 are prose, not scripts**, because log shape drifts across games and SDK versions, so a grep-counting script false-PASSes on another game. The extraction steps, verdict rules (incl. the SmartFloors group-difference exemptions — see `references/smartfloors-user-groups.md`), the mandatory n=5 caveat, and the optional baseline fidelity anchor live in `skills/ad-log-monitor/SKILL.md`. The skill emits no JSON (no orchestrator calls it).

## Design principles

Two principles drive most decisions here — preserve both when extending the plugin.

**Prose first; script only what an agent can't do.** Verification and checking live in agent prose — the agent reads the project and reasons about it. A `scripts/*.sh` is justified **only** for work an agent can't do in prose (locating the plugin root, downloading the SDK, capturing device logs, launching the Unity compiler) — deterministic filesystem/network/toolchain touches. When adding behavior, default to prose and keep it **simple**, stating each check as a behavioral question rather than re-encoding grep patterns. Trade-off (since `2.0.0`): prose verdicts aren't golden-tested, so correctness rests on clear instructions + the user's review.

**Generated code conforms to the host game, not to us.** Codegen adapts to the project — namespace (dominant + `.Metica`, or none), detected adapter folder, existing Max ad-unit IDs, and a remote-config-provider-tailored Step 7 recipe. The canonical *shape* lives in the single `scripts/templates/standalone/MeticaAdService.cs.tmpl`: edit the template, never hand-write C# in the agent prompt. The conform-to-project mechanics live in `integrator.md` (Steps 2.5 and 5) — keep this file pointing there, not restating them.

## Key conventions

- **Instructions describe what the code does — not what it doesn't do or used to do.** These `.md` files are prompts and contributor docs; keep them lean. Don't narrate removed features, prior versions, or absent capabilities ("there is no X anymore", "the old Y", "replaces the old Z", "was an agent through vN"). State current behavior positively. (Imperative do/don't *rules* that govern agent behavior — e.g. "never edit `Assets/MaxSdk/`", "never emit `Metica.AbTest`" — are instructions, not history; keep those.)
- **`metica-versions.yaml` is the single source of truth** for the compatibility matrix. The compat-checker reads it. Add a new SDK by adding an entry and bumping `latest:`. `metica-versions.dev.yaml` is a gitignored local override.
- **Avoid comment/string false positives when scanning `.cs`.** The agents grep to locate a candidate, then Read the surrounding lines to confirm it's live code (not a `//` comment or a `"string literal"`) before acting on it — the agent sees the context.
- **The integrator git-tags `pre-metica-integration`** before any file write (inline `git tag`, after a `git status --porcelain` dirty-tree check — no `git-snapshot.sh` script). On a validator FAIL the integrator runs an **autofix loop** (integrator.md Step 6.5: classify each FAIL autofix/prompt/surface, edit with an anchor re-check, re-validate, max 3 iterations); `git reset --hard pre-metica-integration` is the last-resort rollback **hint** the integrator prints (never auto-executes) only when the loop can't clear all FAILs.
- **Don't mention Linear tickets in repo prose.** Agent prompts, contributor docs, and other committed `.md`/`.tmpl`/agent-facing content describe *what the code does*, not *which ticket motivated it*. A reference like "the MET-XXXXX issue #1 regression" inside an agent's runtime instructions ties this repo to an internal tracker the agent and outside readers can't see, and clutters prose that should read positively. Put the ticket reference in the PR description / commit message / Linear comment instead — those are the right venues for "why we changed this." Existing references in `metica-versions.yaml` / `references/metica-sdk-migration.md` / `agents/unity-compat-checker.md` describe **SDK release artifacts** (which Metica SDK release fixed which issue) — that's a different concern (SDK history, user-facing in upgrade reports) and is grandfathered.
- **Don't name specific customers, studios, games, or their private codebases in repo prose.** Committed `.md`/`.tmpl`/agent-facing content is public-facing plugin source; when real-world evidence motivates a rule or pattern, cite it **anonymously** ("a shipped game", "two shipped games", "a real Metica integration") — never by studio/game name, and never with file paths or `file:line` references into a client's private repository. Same spirit as the Linear-ticket rule: don't tie repo prose to things outside readers can't (and shouldn't) see. The anonymized "two shipped games" phrasing already in the prose is the established pattern.
- **Don't commit planning / scratch artifacts produced by skills (e.g. `superpowers:writing-plans`).** Plans live in the PR description or in the conversation that produced them, not as files in the repo. Keep `docs/` for genuine, hand-curated contributor docs; reject committing anything under `docs/superpowers/`, `docs/plans/`, or similar paths. `.gitignore` covers the common ones.
- **Bump the plugin version (semver) once per PR — never twice.** The release version lives in **two** files that must stay in lockstep: `.claude-plugin/plugin.json` (`version`) and `.claude-plugin/marketplace.json` (`metadata.version`) — never let them drift. A PR bumps the version exactly once, on the commit that introduces the user-visible change; follow-up commits in the same PR (review fixes, CI fixes, docs touch-ups, even substantive redesigns of the same feature) do **not** re-bump. The level is picked from the *cumulative* user-visible impact of the whole PR, not the individual commit: **major** = a break for users or integrators (renaming/removing an agent, changing how agents are invoked or installed, or a breaking change to a sub-agent's JSON contract); **minor** = backward-compatible new capability (a new agent, new script behavior/flag, or a new sub-agent JSON field/rule); **patch** = bug fixes, refactors, or prose/doc tweaks with no behavior change. If a follow-up commit's scope outgrows the bump you already made (e.g. you patch-bumped, then a reviewer asks for a feature addition), edit the existing bump up to the right level rather than stacking a second bump. The plugin release version is the **only** semver in the repo.
