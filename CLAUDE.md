# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

This is **not application code** ‚Äî it is a Claude Code *plugin* that ships four Metica sub-agents: three for Unity integration (compat-checker, integrator, validator) plus a runtime ad-log monitor (`ad-log-monitor`) for on-device verification of any Metica integration. The "source" is:

- **Agent definitions** ‚Äî markdown-with-frontmatter directly under `agents/` (`unity-integrator.md`, `unity-validator.md`, `unity-compat-checker.md`, `ad-log-monitor.md`). The frontmatter (`name`, `description`, `tools`, `model`) is the agent's contract; the body is its prompt. They live in `agents/` itself, **not** a subfolder, on purpose: a plugin subfolder becomes a scope segment in the mention token, so `agents/unity/unity-integrator.md` would register as `@agent-metica-sdk-agents:unity:unity-integrator`. Keeping them flat preserves the documented `@agent-metica-sdk-agents:unity-integrator`.
- **Scripts** ‚Äî `scripts/*.sh` hold all deterministic logic. Agents are thin: they run a script and relay its output. Editing behavior almost always means editing a script, not prose.
- **Reference docs** ‚Äî `agents/contracts.md` (the inter-agent JSON contracts) and `references/` (MaxSdk‚ÜîMeticaSdk API parity).

There is no build step and no compiled artifact. Changing an agent = editing its `.md`; changing logic = editing a `.sh` and its tests.

## Commands

```bash
bash tests/run-all.sh                      # full suite; prints "ALL GREEN" or "FAILURES"
bash tests/run-<suite>-tests.sh            # a single suite (compat, format, download,
                                           # validator, citation, semantic, compile,
                                           # input-validation, codegen-validator, autofix,
                                           # log-monitor)
```

Tests are plain bash assertions against fixtures (`tests/fixtures/`, `tests/validator-fixtures/`) and goldens (`tests/goldens/`). No framework, no install. Suites that probe a sibling `../max-agent-test/DemoApp` skip silently when it's absent ‚Äî a clean clone runs the synthetic-fixture suites only.

## Architecture

**Orchestrator + one thin wrapper + one two-phase validator.** `unity-integrator` (model: sonnet) is the only agent users invoke directly. It calls `unity-compat-checker` (haiku) and `unity-validator` (sonnet) as sub-agents. The **compat-checker is still a thin wrapper** ‚Äî it does nothing but run a script and print its stdout verbatim, so never paraphrase or summarize its output when editing it. The **validator is no longer thin** (as of `validator/1.2.0`): it runs the deterministic floor script (`validate-integration.sh`, `engine: "grep"`) *and* an in-context **semantic-adjudication** phase (`engine: "llm-adjudicator"`) where it reads the project's code and reasons about behaviors grep can't see (indirect reload loops, ready guards across call paths, placement-ID consistency), then merges both into one JSON block. Its grep floor stays golden-tested; its semantic verdicts are evidence-cited and verified by `scripts/check-citation.sh` (the one deterministic, unit-tested piece of the semantic path ‚Äî `tests/run-citation-tests.sh`). The validator runs in a fresh context (the clean room the semantic review needs) and still emits exactly one final ` ```json ` block. See `agents/unity-validator.md`.

**Agents talk to the orchestrator in versioned JSON.** Each sub-agent's final message ends with a fenced ` ```json ` block. The orchestrator parses **the last** such block (regex in `agents/contracts.md`), never the prose. Schemas are versioned `<name>/<major>.<minor>.<patch>`; the orchestrator accepts any minor/patch within an accepted major. **If you add a field to a script's JSON output, bump the minor and update `agents/contracts.md` ‚Äî these three move together.** Contracts: `compat-checker/1.x`, `validator/1.x`. The integrator itself emits no JSON. (`mode-detect/2.x` was retired in v1.0 ‚Äî see "Retired contracts" in `agents/contracts.md`.)

**`PLUGIN_DIR` resolution is mandatory.** Every agent's first bash command is a loop that locates `scripts/resolve-plugin-dir.sh` across known install locations (incl. the marketplace cache `~/.claude/plugins/cache/*/metica-sdk-agents/*`) and runs it. `$CLAUDE_PLUGIN_ROOT` is **not** reliably exported into an agent's bash tool environment ‚Äî verified empirically ‚Äî so it is only the first candidate, never the sole path. `resolve-plugin-dir.sh`'s load-bearing fallback is **self-location** from its own script path (`<root>/scripts/resolve-plugin-dir.sh` ‚Üí root is two levels up); after env vars and self-location it tries agent-symlink targets and known install paths (incl. the version-sorted cache glob). The resolver and the agents' bootstrap loop are covered by `tests/run-resolver-tests.sh`. Scripts are always invoked as `"$PLUGIN_DIR/scripts/..."` ‚Äî never with relative paths. `resolve-plugin-dir.sh`'s `is_root()` identifies the root by the presence of **`.claude-plugin/plugin.json`** and an `agents/` directory; if you move the manifest, that check (and marketplace install) breaks.

**The integrator's flow is discover ‚Üí adapt ‚Üí validate ‚Üí autofix** (RFC v1.0). There is no mode and no `detect-mode.sh`: Step 2 *discovery* derives **whether MaxSDK is present** (`HAS_MAX` = any cleaned `MaxSdk.` reference, the same signal the validator's checks are agnostic to). That fact selects only the mediation argument (`MAX` when present, `null` when absent) and whether the game's direct call sites get rewritten; the generated artifacts are otherwise identical. Discovery (Step 2 + Step 2.5) also inventories direct call sites, the wrapper (a flow-based prose judgment confirmed in plan mode), formats, placements, and triggers into a structured Markdown block (not JSON), then adapts codegen to them via named post-template patch passes (integrator.md Step 5). When Max is present, the integrator's Step 2.5 also detects the project's remote-config provider ‚Äî but the result is **report-only**, driving the Step 7 cohort-gating recipe (a copy-paste pattern the user wires themselves). It does **not** change the generated artifacts: both Max-present cases produce the same `MeticaAdService` + per-format files, and the integrator rewrites the game's *direct* `MaxSdk.*` call sites to use them. Dedicated Max-wrapper files (e.g. an `AdManager.cs` that wraps MaxSDK behind a non-Max API) are left untouched ‚Äî see the wrapper-scoping rule in integrator.md Step 5. **Generated Metica code is a single `MeticaAdService` MonoBehaviour** with one `// @fmt-begin:<format>`‚Ä¶`// @fmt-end:<format>` region per ad format (codegen drops the regions for unused formats; for a larger project the integrator may split each region into a `MeticaAdService.<Format>.cs` `partial class`). It mirrors the docs example: `Start()` calls an idempotent `Initialize()` that sets privacy + `SetLogEnabled` (both before `MeticaSdk.Initialize`, same file), then calls `MeticaSdk.Initialize(config, <mediation>, OnInitialized)` with a **named `OnInitialized(MeticaInitResponse)` method** (not an inline lambda) that logs the `SmartFloors` group/userId and wires up each used format's region. Each region owns its callbacks (named methods, not lambdas ‚Äî extend by adding lines); interstitial/rewarded carry auto-reload-on-hidden + `OnAdShowFailed`-recovery, `IsReady`-guarded show, and **docs-verbatim exponential-backoff retry on load failure** (`Math.Pow(2, Math.Min(6, attempt))` ‚Üí 2‚Üí4‚Üí8‚Ä¶‚Üí64s); banner/MRec carry `OnApplicationFocus` pause/resume + an `_‚Ä¶Showing` state flag and no app-side retry (the SDK refreshes them internally). It exposes per-format `Load*`/`Show*`/`Hide*` delegators for the game. The one template lives at `scripts/templates/standalone/MeticaAdService.cs.tmpl`. The namespace defaults to `<dominant>.Metica` when the project has a dominant namespace, no wrapper at all when the project doesn't use namespaces, and the neutral `MeticaIntegration` when namespaces exist but none dominate ‚Äî **never** `Metica.AbTest` (that label is reserved for the plugin templates' placeholder). Adapters land under `Assets/Scripts/Metica/` by default.

**MeticaSDK install is enforced, never performed.** The integrator does not download or import the SDK. The compat-checker's `metica_sdk` row BLOCKs if it's missing; the user imports it once, then re-runs. `scripts/download-metica-sdk.sh` exists only as a helper the integrator may *offer*.

**The validator's `compiles_cleanly` rule actually builds the project.** `validate-integration.sh` shells out to `scripts/compile-check.sh`, which compiles the whole Unity project in batch-mode (the only compiler that sees Unity's assemblies/asmdefs/defines ‚Äî we deliberately do **not** fall back to `csc`/`dotnet`, which would drown real errors in missing-`UnityEngine` noise) and emits one `compiles_cleanly` FAIL per `error CS####` with file:line. This is the catch-all for compile bugs (it is what would have caught issue #8's unqualified nested enum + wrong property casing), so resist re-adding bespoke per-bug string rules. It is **on by default** but self-skips to a non-blocking `WARN` when no Unity editor is located (`UNITY_PATH` overrides the version-derived Unity Hub search) ‚Äî and **every test suite that invokes the validator exports `METICA_SKIP_COMPILE=1`** so synthetic fixtures never launch Unity. The script's arg/parse/skip/error paths are covered by `tests/run-compile-tests.sh` via a fake-Unity binary; keep that green when touching either file.

**The `ad-log-monitor` agent verifies a Metica + AppLovin MAX integration from live device logs, but its analysis is agent prose, not a deterministic script.** Three phases: **Phase 1** (`scripts/log-monitor-start.sh`) gates the toolchain (`adb` / `idevicesyslog` must be on PATH, hard BLOCK with install hint), detects platform from `adb devices` / `idevice_id -l`, launches a background capture into `./<label>-<platform>-<YYYYMMDDThhmmssZ>.log` (timestamp embedded so re-running with the same label doesn't clobber an earlier log), runs post-launch health checks (PID alive + non-empty file + first line isn't a tool error), and writes `./<label>.session` (plain `key=value`, parsed ‚Äî **never sourced** ‚Äî by stop.sh, with the `pid` and `label` fields validated before any `kill` so a tampered file can't be weaponised). **Phase 2** is two beats: **2a** is `scripts/log-monitor-stop.sh` ‚Äî kills the capture, prints a small summary (label / platform / log path / line count / app filter / start time), removes the session file, and exits. It does **no** analysis. **2b is the agent**: it reads the captured log, runs targeted greps per the steps in `agents/ad-log-monitor.md` (stack inventory, per-format extraction with ad unit IDs / lifecycle / network attribution / revenue per impression, Metica‚ÜíMAX floor handoff, load strategy from timestamps, errors), interprets the lines it sees, and writes `./<label>-analysis.md`. **The reason Phase 2b is prose, not a script:** log shape varies between games and SDK versions (class names, prefixes, log levels, custom wrappers, AppLovin internal state-machine strings drift across MAX releases), so a grep-counting script written against one game's log produces false PASS/FAIL on another. This mirrors the approach of the user-level `monitor-ad-logcat` skill, which the agent absorbed and extended (the agent adds iOS support, the `<label>.session` capture handoff, and Phase 3). **Phase 3** is **entirely agent prose** ‚Äî reads both per-route analyses + both raw logs, builds the side-by-side delta table, applies verdict rules (`trial revenue/impression < holdout` ‚Üí flag; trial-only lifecycle anomalies ‚Üí flag; trial-only errors ‚Üí flag), and writes `./compare-trial-vs-holdout.md`. The mandatory **n=5 caveat** stays in the verdict prose ‚Äî at five impressions per format the agent can spot directional regressions but cannot declare a revenue winner. The agent emits **no JSON** to an orchestrator (it isn't called by one), so it has no entry in `agents/contracts.md`.

## Design principles

Two principles drive most decisions here ‚Äî preserve both when extending the plugin.

**Balance prose against scripting.** Deterministic, repeatable logic goes in a `scripts/*.sh` with a test (and a golden where the output is fixed) ‚Äî that's why compat detection, validation, and report formatting are scripts fronted by thin agents. Logic that is a one-off judgment the user reviews anyway stays in the agent prose, intentionally unscripted: the integrator's namespace and remote-config detection (integrator.md Step 2.5) runs inline in the agent ‚Äî not as a tested script ‚Äî *because* "perfect precision isn't required; the user approves the detected value in the plan." `ad-log-monitor`'s Phase 2b is the same call: log shape varies enough between games / SDK versions that a grep-counting analyzer gives false PASS/FAIL, so the agent runs targeted greps and interprets the lines itself rather than locking goldens. When adding behavior, place it on the correct side: don't script a judgment call, and don't bury testable logic in a prompt where it can't be covered by a golden.

**Generated code conforms to the host game, not to us.** Codegen adapts to the project it lands in ‚Äî it wraps files in the project's dominant namespace (+ `.Metica`), or omits the namespace wrapper entirely when the project doesn't use namespaces; uses the detected adapter folder; reuses existing Max ad unit IDs; and (when a remote-config provider is detected) tailors the Step 7 cohort-gating recipe to that provider rather than emitting a router/binding. The canonical *shape* of the generated code lives in the single `scripts/templates/standalone/MeticaAdService.cs.tmpl` (one MonoBehaviour, per-format `@fmt` regions): edit the template, never hand-write C# in the agent prompt. The conform-to-project mechanics live in integrator.md (Steps 2.5 and 5) ‚Äî keep this file pointing there rather than restating them, so the two can't drift.

## Key conventions

- **`metica-versions.yaml` is the single source of truth** for the compatibility matrix. The compat-checker reads it. Add a new SDK by adding an entry and bumping `latest:`. `metica-versions.dev.yaml` is a gitignored local override.
- **String-literal-aware grepping.** Source scans pipe `grep` through `scripts/lib/clean-cs.awk` to ignore matches inside C# comments and string literals. Reuse it rather than raw `grep` when scanning `.cs`.
- **The integrator git-tags `pre-metica-integration`** before any file write (`scripts/git-snapshot.sh`). On a validator FAIL the integrator runs an **autofix loop** (integrator.md Step 6.5: classify each FAIL autofix/prompt/surface, edit with an anchor re-check, re-validate, max 3 iterations); `git reset --hard pre-metica-integration` is the last-resort rollback **hint** the integrator prints (never auto-executes) only when the loop can't clear all FAILs.
- **Bump the plugin version (semver) once per PR ‚Äî never twice.** The release version lives in **two** files that must stay in lockstep: `.claude-plugin/plugin.json` (`version`) and `.claude-plugin/marketplace.json` (`metadata.version`) ‚Äî never let them drift. A PR bumps the version exactly once, on the commit that introduces the user-visible change; follow-up commits in the same PR (review fixes, CI fixes, docs touch-ups, even substantive redesigns of the same feature) do **not** re-bump. The level is picked from the *cumulative* user-visible impact of the whole PR, not the individual commit: **major** = a break for users or integrators (renaming/removing an agent, changing how agents are invoked or installed, or a breaking JSON-contract *major* bump like `compat-checker/2.x`); **minor** = backward-compatible new capability (a new agent, new script behavior/flag, or a new JSON-contract field ‚Äî which already triggers a contract *minor* bump, see Architecture); **patch** = bug fixes, refactors, or prose/doc tweaks with no behavior change. If a follow-up commit's scope outgrows the bump you already made (e.g. you patch-bumped, then a reviewer asks for a feature addition), edit the existing bump up to the right level rather than stacking a second bump. This plugin release version is **distinct** from the per-contract schema versions in `agents/contracts.md` ‚Äî the two move independently.
