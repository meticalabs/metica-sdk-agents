# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

This is **not application code** — it is a Claude Code *plugin* that ships three Metica sub-agents (compat-checker, integrator, validator) for Unity integration, plus an `ad-log-monitor` skill for on-device runtime verification. The "source" is:

- **Agent definitions** — markdown-with-frontmatter directly under `agents/` (`unity-integrator.md`, `unity-validator.md`, `unity-compat-checker.md`). The frontmatter (`name`, `description`, `tools`, `model`) is the agent's contract; the body is its prompt. They live in `agents/` itself, **not** a subfolder, on purpose: a plugin subfolder becomes a scope segment in the mention token, so `agents/unity/unity-integrator.md` would register as `@agent-metica-sdk-agents:unity:unity-integrator`. Keeping them flat preserves the documented `@agent-metica-sdk-agents:unity-integrator`.
- **Skill definitions** — `skills/<name>/SKILL.md` (auto-discovered by Claude Code from the `skills/` directory; no entry in `plugin.json` needed). The frontmatter is just `name:` + `description:` (skills inherit the conversation's model and tool set, so `tools:` and `model:` aren't applicable). Invoked via the `/` slash form (`/metica-sdk-agents:<skill-name>`) or a description-matched trigger phrase — **not** the `@`-mention syntax used for agents (that's agents-only).
- **Scripts** — `scripts/*.sh` hold only the few things an agent **can't** do in prose: locating the plugin root (`resolve-plugin-dir.sh`), downloading + importing the SDK (`download-metica-sdk.sh`), capturing device logs (`log-monitor-start.sh` / `log-monitor-stop.sh`), launching the Unity compiler (`compile-check.sh`), and the SessionStart update-notify check (`check-for-update.sh`). The verification/checking logic (compat detection, integration validation, report formatting) is **not** scripted anymore — it lives in the agent prose, which reads the project and reasons about it. Editing behavior almost always means editing an agent's `.md`, not a script.
- **Hooks** — `hooks/hooks.json` (auto-discovered by Claude Code from the `hooks/` directory; no entry in `plugin.json` needed). It wires one `SessionStart` command hook → `scripts/check-for-update.sh`, which compares the installed version to the latest published `.claude-plugin/plugin.json` on the repo's default branch and, only when the remote is strictly newer, prints a `SessionStart` `additionalContext` notice with the `/plugin marketplace update metica-sdk-agents` command. It is **fail-open**: any uncertainty (no `curl`, no network, a 404, an unparseable version) exits 0 with no output, so it can never block or nag a session start. Opt out with `METICA_SKIP_UPDATE_CHECK=1`; `METICA_UPDATE_URL` / `METICA_UPDATE_TIMEOUT` override the source/timeout (used by the tests).
- **Data + templates** — `metica-versions.yaml` (compat matrix), `references/max-metica-api-map.tsv` (+ its narrative twin) and `scripts/templates/standalone/MeticaAdService.cs.tmpl` are read directly by the agents.
- **Reference docs** — `agents/contracts.md` (the inter-agent JSON contracts) and `references/` (MaxSdk↔MeticaSdk API parity).

There is no build step and no compiled artifact. Changing an agent or skill = editing its `.md`; the surviving scripts change only when the system tooling around them does.

## Commands

```bash
bash tests/run-all.sh                      # full suite; prints "ALL GREEN" or "FAILURES"
bash tests/run-<suite>-tests.sh            # a single suite: resolver, download,
                                           # compile, update-check, log-monitor
```

Tests now cover only the **surviving scripts** (the system-tooling plumbing) — there are five
suites. The verification logic moved into agent prose, which is reviewed by the user at run
time rather than golden-tested, so the old `compat` / `format` / `validator` / `citation` /
`semantic` / `codegen` / `autofix` / `input-validation` suites and their fixtures/goldens were
removed. Tests are plain bash assertions against `tests/fixtures/` (now just `log-monitor/`).
No framework, no install. The `download` suite skips silently when its local SDK build
(`../Metica SDK builds/MeticaSdk-2.4.0.unitypackage`) is absent; the other four always run.
The `update-check` suite drives `check-for-update.sh` through a fake `curl` (no network).

## Architecture

**Orchestrator + two reasoning sub-agents.** `unity-integrator` (model: sonnet) is the only agent users invoke directly. It calls `unity-compat-checker` (haiku) and `unity-validator` (sonnet) as sub-agents. Both sub-agents are **reasoning agents, not script-wrappers** (as of `2.0.0`): the **compat-checker** reads the project's marker files + `metica-versions.yaml` and judges each check in prose, then emits one `compat-checker/1.0.0` JSON block; the **validator** reads the project's code and reasons about every rule in a single pass — structural rules (init count, privacy ordering, callback parity) and behavioral ones grep gets wrong (indirect reload loops, ready guards across call paths, placement-ID consistency) — citing the lines that prove each verdict, then emits one `validator/2.1.0` JSON block. The validator's only shell-out is the Unity batch compile (`compile-check.sh`) behind `compiles_cleanly`; it verifies its own citations by re-reading the file (there is no `check-citation.sh` anymore). Both run in a fresh context (the clean room the review needs) and emit exactly one final ` ```json ` block. See `agents/unity-validator.md` and `agents/unity-compat-checker.md`.

**Agents talk to the orchestrator in versioned JSON.** Even though the sub-agents now reason in prose, each still **ends with one fenced ` ```json ` block** so the orchestrator parses a stable machine signal, not the prose. The orchestrator parses **the last** such block (regex in `agents/contracts.md`). Schemas are versioned `<name>/<major>.<minor>.<patch>`; the orchestrator accepts any minor/patch within an accepted major. **If you change a sub-agent's JSON output, update `agents/contracts.md` in the same commit and bump the schema version.** Contracts: `compat-checker/1.x`, `validator/2.x`. The integrator itself emits no JSON. (`mode-detect/2.x` was retired in v1.0 — see "Retired contracts" in `agents/contracts.md`.)

**`PLUGIN_DIR` resolution is mandatory.** Every agent's (and the skill's) first bash command is a loop that locates `scripts/resolve-plugin-dir.sh` across known install locations (incl. the marketplace cache `~/.claude/plugins/cache/*/metica-sdk-agents/*`) and runs it. This loop is the **one block that must stay literal bash** and is **deliberately duplicated, byte-identical, across all four entry points** (the three agents + `skills/ad-log-monitor/SKILL.md`) — it can't be DRY'd into a shared script because its whole job is to *find* the scripts before any path to them is known (the bootstrap paradox). When you change it, change all four copies in lockstep. `$CLAUDE_PLUGIN_ROOT` is **not** reliably exported into an agent's bash tool environment — verified empirically — so it is only the first candidate, never the sole path. `resolve-plugin-dir.sh`'s load-bearing fallback is **self-location** from its own script path (`<root>/scripts/resolve-plugin-dir.sh` → root is two levels up); after env vars and self-location it tries agent-symlink targets and known install paths (incl. the version-sorted cache glob). The resolver and the agents' bootstrap loop are covered by `tests/run-resolver-tests.sh`. Scripts are always invoked as `"$PLUGIN_DIR/scripts/..."` — never with relative paths. `resolve-plugin-dir.sh`'s `is_root()` identifies the root by the presence of **`.claude-plugin/plugin.json`** and an `agents/` directory; if you move the manifest, that check (and marketplace install) breaks.

**The integrator's flow is discover → adapt → validate → autofix.** There is no mode and no `detect-mode.sh`: Step 2 *discovery* derives **whether MaxSDK is present** (`HAS_MAX` = any real `MaxSdk.` reference — grep to locate, Read to confirm it's live code, not a comment/string — the same signal the validator's checks are agnostic to). That fact selects only the mediation argument (`MAX` when present, `null` when absent) and whether the game's direct call sites get rewritten; the generated artifacts are otherwise identical. Discovery (Step 2 + Step 2.5) also inventories direct call sites, the wrapper (a flow-based prose judgment confirmed in plan mode), formats, placements, triggers, and **per-format behaviour** (load trigger, show trigger, any frequency gate/cooldown between shows, and single-vs-multiple ad units — so codegen preserves the game's existing ad cadence rather than just swapping the API) into a structured Markdown block (not JSON), then adapts codegen to them via named post-template patch passes (integrator.md Step 5). When Max is present, the integrator's Step 2.5 also detects the project's remote-config provider — but the result is **report-only**, driving the Step 7 cohort-gating recipe (a copy-paste pattern the user wires themselves). It does **not** change the generated artifacts: both Max-present cases produce the same `MeticaAdService` + per-format files, and the integrator rewrites the game's *direct* `MaxSdk.*` call sites to use them. Dedicated Max-wrapper files (e.g. an `AdManager.cs` that wraps MaxSDK behind a non-Max API) are left untouched — see the wrapper-scoping rule in integrator.md Step 5. **Generated Metica code is a single `MeticaAdService` MonoBehaviour** with one `// @fmt-begin:<format>`…`// @fmt-end:<format>` region per ad format (codegen drops the regions for unused formats; for a larger project the integrator may split each region into a `MeticaAdService.<Format>.cs` `partial class`). It mirrors the docs example: `Start()` calls an idempotent `Initialize()` that sets privacy + `SetLogEnabled` (both before `MeticaSdk.Initialize`, same file), then calls `MeticaSdk.Initialize(config, <mediation>, OnInitialized)` with a **named `OnInitialized(MeticaInitResponse)` method** (not an inline lambda) that logs the `SmartFloors` group/userId and wires up each used format's region. Each region owns its callbacks (named methods, not lambdas — extend by adding lines); interstitial/rewarded carry auto-reload-on-hidden + `OnAdShowFailed`-recovery, `IsReady`-guarded show, and **docs-verbatim exponential-backoff retry on load failure** (`Math.Pow(2, Math.Min(6, attempt))` → 2→4→8…→64s); banner/MRec carry `OnApplicationFocus` pause/resume + an `_…Showing` state flag and no app-side retry (the SDK refreshes them internally). It exposes per-format `Load*`/`Show*`/`Hide*` delegators for the game. The one template lives at `scripts/templates/standalone/MeticaAdService.cs.tmpl`. The namespace defaults to `<dominant>.Metica` when the project has a dominant namespace, no wrapper at all when the project doesn't use namespaces, and the neutral `MeticaIntegration` when namespaces exist but none dominate — **never** `Metica.AbTest` (that label is reserved for the plugin templates' placeholder). Adapters land under `Assets/Scripts/Metica/` by default.

**MeticaSDK install is enforced, never performed.** The integrator does not download or import the SDK. The compat-checker's `metica_sdk` row BLOCKs if it's missing; the user imports it once, then re-runs. `scripts/download-metica-sdk.sh` exists only as a helper the integrator may *offer*.

**`references/max-metica-api-map.tsv` is the single source of truth for the MaxSdk ↔ MeticaSdk surface.** Both the validator (rules `max_api_use_metica` and `max_api_unsupported`, read in prose) and the integrator (Step 5 rewrite + drop pass) read the same TSV, so they can't drift. Rows are `<pattern>\t<replacement>\t<kind>\t<notes>` where `kind` is `rename` (direct swap), `signature-change` (replace + adjust caller — e.g. `SetBannerBackgroundColor` switches from `UnityEngine.Color` to a hex string), `drop` (no Metica equivalent — remove the call and surface in the report), or `exempt` (`MaxSdkUtils.*`, stateless helpers, mix-safe). Validator file-context split: matches in a file that also references `MeticaSdk.` emit **FAIL** (the file is part of the Metica integration; the `MaxSdk.*` call is dead because Metica owns init); matches in a pure-Max file (e.g. a side-by-side `AdManager.cs` wrapper) emit **ADVISORY** (the calls still no-op under Metica's init, but the wrapper might be intentionally kept). The narrative form of the same mapping lives at `references/max-vs-metica-2.4.0-api.md`; **keep both in sync when the SDK bumps**.

**The validator's `compiles_cleanly` rule actually builds the project.** The validator agent shells out to `scripts/compile-check.sh`, which compiles the whole Unity project in batch-mode (the only compiler that sees Unity's assemblies/asmdefs/defines — we deliberately do **not** fall back to `csc`/`dotnet`, which would drown real errors in missing-`UnityEngine` noise) and prints tab-delimited records the agent turns into `compiles_cleanly` findings (one FAIL per `error CS####` with file:line). This is the catch-all for compile bugs (it is what would have caught issue #8's unqualified nested enum + wrong property casing), so resist re-adding bespoke per-bug string rules in the prose. It is **on by default** but self-skips to a non-blocking `WARN` when no Unity editor is located (`UNITY_PATH` overrides the version-derived Unity Hub search) or when `METICA_SKIP_COMPILE=1` is set. The script's arg/parse/skip/error paths are covered by `tests/run-compile-tests.sh` via a fake-Unity binary; keep that green when touching the script.

**The `ad-log-monitor` skill verifies a Metica + AppLovin MAX integration from live device logs.** It is a **skill, not an agent** (as of v1.5.0 — was an agent through v1.4.0). The conversion is deliberate: the workflow is interactive (start → user plays → stop → analyse → repeat for second route → compare) and benefits from staying in the main conversation, where the user can interject questions on the analysis without context-switching to a sub-agent. The other Metica components stay as agents because they do isolated codegen / validation work where fresh-context isolation is the point. Three phases: **Phase 1** (`scripts/log-monitor-start.sh`) gates the toolchain (`adb` / `idevicesyslog` must be on PATH, hard BLOCK with install hint), detects platform from `adb devices` / `idevice_id -l`, launches a background capture into `./<label>-<platform>-<YYYYMMDDThhmmssZ>.log` (timestamp embedded so re-running with the same label doesn't clobber an earlier log), runs post-launch health checks (PID alive + non-empty file + first line isn't a tool error), and atomically claims `./<label>.session` (plain `key=value`, parsed — **never sourced** — by stop.sh, with the `pid` and `label` fields validated before any `kill` so a tampered file can't be weaponised). **Phase 2** is two beats: **2a** is `scripts/log-monitor-stop.sh` — kills the capture, prints a small summary (label / platform / log path / line count / app filter / start time), removes the session file, and exits. It does **no** analysis. **2b is the skill**: it reads the captured log, runs targeted greps per the steps in `skills/ad-log-monitor/SKILL.md` (stack inventory, per-format extraction with ad unit IDs / lifecycle / network attribution / revenue per impression, Metica→MAX floor handoff, load strategy from timestamps, errors), interprets the lines it sees, and writes `./<label>-analysis.md`. **The reason Phase 2b is prose, not a script:** log shape varies between games and SDK versions (class names, prefixes, log levels, custom wrappers, AppLovin internal state-machine strings drift across MAX releases), so a grep-counting script written against one game's log produces false PASS/FAIL on another. This mirrors the approach of the user-level `monitor-ad-logcat` skill, which this skill absorbed and extended (adds iOS support, the `<label>.session` capture handoff, and Phase 3). **Phase 3** is **entirely prose** — reads both per-route analyses + both raw logs, builds the side-by-side delta table, applies verdict rules (`trial revenue/impression < holdout` → flag; trial-only lifecycle anomalies → flag; trial-only errors → flag), and writes `./compare-trial-vs-holdout.md`. The mandatory **n=5 caveat** stays in the verdict prose — at five impressions per format the skill can spot directional regressions but cannot declare a revenue winner. An **optional `baseline` route** (the live App Store / Play Store build) can be captured alongside the two dev-build routes; Phase 3 treats it strictly as a **fidelity anchor** — does the holdout dev-build reproduce production? — never as a third A/B arm or the pass/fail gate (it's a different build: store signing, possibly a different version, likely no Metica). Holdout + trial are developer-provided dev builds; baseline comes from the store. The trial-vs-holdout verdict is unchanged whether or not a baseline is present. The skill emits **no JSON** to an orchestrator (it isn't called by one), so it has no entry in `agents/contracts.md`.

## Design principles

Two principles drive most decisions here — preserve both when extending the plugin.

**Prose first; script only what an agent can't do.** Verification and checking live in the agent prose — the agent reads the project and reasons about it (compat detection, integration validation, the Max-API rewrite/flag decisions, log analysis are all prose). A `scripts/*.sh` is justified **only** when the work is genuinely impossible or unwise for an agent to do in prose: locating the plugin root, downloading + importing the SDK, capturing device logs, and launching the Unity compiler. Those few stay scripted (and tested), because they touch the filesystem/network/toolchain deterministically and an LLM can't "reason" a compile into existence. Everything else is prose the user reviews at run time. When adding behavior, default to prose; reach for a script only when you hit one of those hard system boundaries — and keep the prose **simple**, stating each check as a behavioral question rather than re-encoding grep patterns. The trade-off accepted in `2.0.0`: prose verdicts aren't golden-tested, so correctness rests on clear instructions + the user's review, not on a fixture suite.

**Generated code conforms to the host game, not to us.** Codegen adapts to the project it lands in — it wraps files in the project's dominant namespace (+ `.Metica`), or omits the namespace wrapper entirely when the project doesn't use namespaces; uses the detected adapter folder; reuses existing Max ad unit IDs; and (when a remote-config provider is detected) tailors the Step 7 cohort-gating recipe to that provider rather than emitting a router/binding. The canonical *shape* of the generated code lives in the single `scripts/templates/standalone/MeticaAdService.cs.tmpl` (one MonoBehaviour, per-format `@fmt` regions): edit the template, never hand-write C# in the agent prompt. The conform-to-project mechanics live in integrator.md (Steps 2.5 and 5) — keep this file pointing there rather than restating them, so the two can't drift.

## Key conventions

- **`metica-versions.yaml` is the single source of truth** for the compatibility matrix. The compat-checker reads it. Add a new SDK by adding an entry and bumping `latest:`. `metica-versions.dev.yaml` is a gitignored local override.
- **Avoid comment/string false positives when scanning `.cs`.** There is no `clean-cs.awk` stripper anymore. The agents grep to locate a candidate, then Read the surrounding lines to confirm it's live code (not a `//` comment or a `"string literal"`) before acting on it — more reliable than grep-with-stripping because the agent sees the context.
- **The integrator git-tags `pre-metica-integration`** before any file write (inline `git tag`, after a `git status --porcelain` dirty-tree check — no `git-snapshot.sh` script). On a validator FAIL the integrator runs an **autofix loop** (integrator.md Step 6.5: classify each FAIL autofix/prompt/surface, edit with an anchor re-check, re-validate, max 3 iterations); `git reset --hard pre-metica-integration` is the last-resort rollback **hint** the integrator prints (never auto-executes) only when the loop can't clear all FAILs.
- **Bump the plugin version (semver) once per PR — never twice.** The release version lives in **two** files that must stay in lockstep: `.claude-plugin/plugin.json` (`version`) and `.claude-plugin/marketplace.json` (`metadata.version`) — never let them drift. A PR bumps the version exactly once, on the commit that introduces the user-visible change; follow-up commits in the same PR (review fixes, CI fixes, docs touch-ups, even substantive redesigns of the same feature) do **not** re-bump. The level is picked from the *cumulative* user-visible impact of the whole PR, not the individual commit: **major** = a break for users or integrators (renaming/removing an agent, changing how agents are invoked or installed, or a breaking JSON-contract *major* bump like `compat-checker/2.x`); **minor** = backward-compatible new capability (a new agent, new script behavior/flag, or a new JSON-contract field — which already triggers a contract *minor* bump, see Architecture); **patch** = bug fixes, refactors, or prose/doc tweaks with no behavior change. If a follow-up commit's scope outgrows the bump you already made (e.g. you patch-bumped, then a reviewer asks for a feature addition), edit the existing bump up to the right level rather than stacking a second bump. This plugin release version is **distinct** from the per-contract schema versions in `agents/contracts.md` — the two move independently.
